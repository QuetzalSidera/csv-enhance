# Roadmap

CSVX 现在已经能完成一条很完整的主干流程：

- 写 `.csvx`
- 做强类型表格
- 生成计算列
- 使用本地 `@func` 和外部 `@plugin`
- 生成窗口列
- 导出 `.xlsx`
- 在 VS Code 中获得基础编辑器支持

接下来的路线，不再只是“把功能往上堆”，而是让 CSVX 逐步变成一个：

- 更好写
- 更好读
- 更好用
- 更适合编辑器和 AI 协作

的格式。

---

## 1. 近期重点

这些是最值得优先推进的方向。

### 1.1 `@plot` 继续打磨

当前 `@plot` 已经可用，但仍然偏“原型版”。

近期目标：

- 支持比 bar 更丰富的图表类型
- 优化 `deps / x / y / color / title` 的书写体验
- 提供更清晰的错误提示和默认行为
- 让 plot 输出对前端消费更稳定

### 1.2 图像渲染

当前 `@plot` 主要输出 Vega-Lite 规格和 `_plots` 元数据。

未来目标：

- 支持将 `@plot` 渲染成真正的图像产物
- 支持导出 PNG / SVG 等可直接展示的结果
- 在 `.xlsx` 导出链路中探索图像嵌入或更完整图表呈现

这部分很有价值，因为它能让 CSVX 从“会描述图”进一步走向“会产出图”。

### 1.3 更丰富、也更顺手的语法

CSVX 的方向不是“变成另一门很重的语言”，而是：

**让常见事情更自然地表达出来。**

近期可以继续打磨：

- 更短、更直觉的 block 写法
- 更友好的默认行为
- 更少样板代码的表达式书写
- 更自然的名字和字段设计

目标不是追求花哨，而是让第一次写 CSVX 的人更少卡壳。

### 1.4 编辑器能力继续升级

VS Code 已经有不错的第一版支持，但还可以继续补强：

- 更智能的 completion
- 更细致的 hover 信息
- 更完整的 go to definition / references
- rename
- quick fix / code action
- 更贴近真实开发体验的诊断展示

---

## 2. 编辑器生态

CSVX 不能只在一个编辑器里好用。

### 2.1 IDEA / JetBrains 编辑器支持

当前主要支持 VS Code。

下一步非常值得做的是：

- IntelliJ IDEA / WebStorm / PyCharm 等 JetBrains 系列编辑器高亮
- 基础 diagnostics 接入
- hover / definition 能力探索

这会让 CSVX 的落地范围更大，也能减少“换编辑器就失去体验”的问题。

### 2.2 通用语言服务

随着 editor 能力继续增长，适合逐步考虑：

- 更稳定的语言服务层
- 更通用的 LSP（Language Server Protocol）支持

这样未来不只是 VS Code 和 IDEA，其他支持 LSP 的编辑器也有机会接入。

---

## 3. 语言能力扩展

这些方向会决定 CSVX 未来能表达多复杂的表格逻辑。

### 3.1 `@window` 继续增强

当前 `@window` 已经支持一批基础窗口函数。

后续可以继续扩展：

- 更丰富的窗口内置函数
- 更灵活的分组与排序能力
- 更清晰的窗口调试体验
- 更适合新手理解的语法糖

### 3.2 更丰富的内置函数

当前已经有：

- `if`
- `coalesce`
- `and`
- `or`
- `lag`
- `lead`
- `cumsum`
- `rank`

后续可以补的方向包括：

- 更多字符串函数
- 更多数字函数
- 更多日期时间函数
- 更方便的数据清洗函数

这会让很多原本必须依赖插件的场景，直接在 `.csvx` 内完成。

### 3.3 更成熟的类型系统

CSVX 已经有了不错的强类型基础，但还可以继续前进：

- 更清晰的 shape/type 文档与行为边界
- 更智能的类型报错
- 更完整的自动推断
- 更好的函数参数与返回值检查

目标是让类型系统既可靠，又不把新手吓跑。

### 3.4 聚合与更高层数据能力

当前已经有 row-scope 的 `@compute` 和 sequence-scope 的 `@window`。

未来还值得探索：

- 更自然的聚合能力
- 更适合统计场景的 block
- 多表能力
- join / merge / reshape 这一类更高阶的数据操作

不过这些能力需要非常小心，不能让 CSVX 失去“轻量”和“易读”。

---

## 4. 输出与集成

CSVX 的价值不只在写文件，也在“最后能接到真实世界”。

### 4.1 更强的 `.xlsx` 导出

当前 `.xlsx` 导出已经可用，但仍有继续提升空间：

- 更完整的 worksheet 格式化
- 更丰富的 metadata sheet
- 更贴近真实电子表格使用场景的结构
- 探索原生图表或图像嵌入

### 4.2 更多输出目标

未来可以探索：

- JSON 导出
- HTML 预览页
- 静态图表产物
- 适合后端或前端继续消费的中间格式

### 4.3 更完整的 CLI 体验

当前 CLI 已经支持核心流程。

后续可以增强：

- 更友好的命令帮助
- 更清晰的输出摘要
- 更方便的批量处理
- 更适合 CI 使用的模式

---

## 5. AI 友好性

CSVX 的一个独特方向，就是它不只是给人写，也希望 AI 写得顺、改得稳。

这部分很值得长期投入。

### 5.1 更稳定的生成模式

目标包括：

- 更少容易写错的样板
- 更少歧义
- 更短但仍可读的表达方式
- 更稳定的 block 结构

### 5.2 更适合 AI 的错误信息

未来可以继续增强：

- 更直接的建议
- 更容易自动修复的诊断
- 更适合模型“看一眼就知道怎么改”的报错风格

### 5.3 Lint 规则继续生长

CSVX 已经有 lint 和 diagnostics 基础设施。

后续可以继续加入：

- 更面向可维护性的规则
- 更面向 AI 生成质量的规则
- 更面向团队规范的规则

## 7. 已知限制

下面这些限制目前仍然存在，后续需要继续打磨。

### 7.1 注释与表格内容的边界

当前以 `#` 开头的整行会被当成注释。

这意味着：

- 如果某一整行 table 内容在去掉首尾空格后以 `#` 开头
- 它会被视为注释，而不是数据

这条规则对大多数文件是方便的，但对特殊数据仍然偏粗糙。

### 7.2 `@plot` 仍然偏原型

虽然已经能工作，但：

- 图表类型还不够丰富
- 渲染链路还不够完整
- 输出体验还需要继续打磨

### 7.3 编辑器支持还不够广

目前以 VS Code 为主。  
JetBrains / IDEA 生态还没有正式补齐。

---

## 8. 一个更长远的目标

CSVX 不只是“再做一种表格格式”。

更长远的方向，其实是让它成为一种：

- 适合人写
- 适合 AI 改
- 适合程序执行
- 适合导出成传统电子表格

的桥梁格式。

如果这件事做成了，CSVX 会很像表格世界里的“中间语言”：

- 轻
- 清楚
- 可扩展
- 可编译

这也是这条路线最有意思的地方。
